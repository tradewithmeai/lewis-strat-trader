## Look-ahead & circularity: does any label use forward returns; is the 7-day novelty feature causal; is the 24h embargo sufficient given overlapping forward-return windows; and is reporting topic_china<0 a circular "finding"? Where does hindsight leak?

**What holds up:** The arithmetic of label construction is genuinely clean, and unusually disciplined for this literature — I verified it in the code, not just the prose:
- No forward return enters any label. The directional signal is built from FinBERT P(pos)-P(neg), the gpt-4.1-mini stance/conviction read, frozen keyword topics, and novelty — all from post text+timestamp only (trump_signal_local.py, trump_classify.py). Confirmed.
- The 7-day novelty feature is strictly causal. In _novelty() (trump_signal_local.py lines 98-116) the similarity is computed only against emb[lo:i] — posts strictly before index i — with a time-windowed lower bound, and the `if i > lo` guard leaves the first post in any window at novelty=1.0 (no peeking at later near-duplicates). The archive is sorted ascending by ts (trump_archive.load(), line 105), so index order == time order. No forward leakage in novelty.
- Forward targets are measured from the EVENT HOUR'S CLOSE (logc.shift(-k) - logc), deliberately discarding the first-minutes reaction — conservative, no contemporaneous-bar contamination.
- trail_vol_24 is shifted by 1 bar (...).shift(1), so the trailing-vol control is strictly known at event time; the endogenous-timing confound ("he posts because markets are already moving") is correctly addressed by the era + trailing-vol-quintile matched null.
- Engagement/favourites are explicitly excluded from all predictive specs because they are not known at post time (used descriptively only). This is exactly the right call and is enforced in the feature list (feat_cols in _phase2_reaction.py).
- The famous DJT "GREAT TIME TO BUY" post is correctly treated as an n=2 case study, not a statistic.
Bottom line on the narrow question "does a return enter a label / is novelty causal / is the event-time anchor leaky": no, yes, no. That part survives.

**Verdict:** On this axis the study is partly publication-credible and partly not. The mechanical look-ahead controls are genuinely sound and verified in code — no forward return enters any label, the 7-day novelty feature is strictly causal, forward targets are anchored at the event-hour close, the trailing-vol control is properly lagged, and engagement is correctly excluded. That discipline is real and uncommon; keep it. BUT the hindsight does not leak through the arithmetic — it leaks upstream, and the manuscript's 'no forward returns in labels' framing gives false comfort against it. Two HIGH issues must be resolved before publication: (1) the taxonomy/features were designed on the same 2025-26 episodes they are tested on, with the motivating April-2025 episode sitting in-sample and before the OOS holdout, and no uncontaminated in-regime test set exists; (2) the LLM labeler can encode the known outcomes of famous posts, contaminating 'text-only' labels through its training weights. Both have cheap, concrete fixes (confirmatory claims for hindsight-derived categories restricted to the verified OOS holdout; a FinBERT-only robustness column). The on-angle embargo point is real but currently moot because the purged CV the paper advertises is not actually implemented. Not yet acceptable on the look-ahead/circularity axis; revise-and-resubmit.

- **[HIGH]** Taxonomy and features were designed on the observed outcomes, and the confirmatory tests then run on the same episodes — pre-registration does not launder this. The paper itself states the taxonomy was 'refined after observing specific 2025 episodes,' and the code makes it explicit: the `market_directive` regex is anchored in a comment to '2025-04-09, 4h before the tariff-pause rally' and `reassurance` to stress-period phrases ('BE COOL', 'don't panic'). These categories are DEFINED by reference to known market-moving events. Pre-registration (frozen 2026-06-02) freezes the specification before the coefficients are computed, which controls spec-mining — but it puts NO temporal firewall between the data that DESIGNED the features and the data being TESTED. I checked where the motivating episode sits: the 2025-04-09 post is at ~73.5% through the Feb-2022→Jun-2026 sample, i.e. it is IN-sample for the full-corpus event study and the full-sample OLS grid, and it falls BEFORE the final-20% OOS holdout (calendar boundary ~2025-07-21; the ordinal-by-event split is even later given the 2025-26 posting surge). So the hindsight-derived categories are evaluated on precisely the data that created them. Compounding this, the authors' own disclosed caveat — 'single regime dominated by the 2025-26 tariff war' — means the ONLY regime containing the phenomena of interest is also the regime used to build the taxonomy: there is no uncontaminated in-regime test set. This, not the label arithmetic, is where hindsight leaks.
  - *Fix:* Restrict any CONFIRMATORY claim for the hindsight-derived categories (market_directive, reassurance, and any taxonomy element refined post-2025) to the final-20% OOS holdout, and explicitly verify the motivating episodes predate the holdout (they do — April 2025 is in-sample), so those episodes cannot count as confirmation. Report all full-sample results for these categories as exploratory/Phase-0-equivalent, sitting alongside the existing exploratory event study, not as pre-registered confirmations. State plainly that the taxonomy-defining regime and the test regime coincide, so the categories are not out-of-regime validated.
- **[HIGH]** The gpt-4.1-mini stance/conviction labels can leak forward knowledge through the model's training corpus, silently violating 'text-only, no forward returns.' An LLM's parametric memory encodes how famous posts played out: the 'GREAT TIME TO BUY' post is famous PRECISELY because of the subsequent 90-day-pause rally, and 4.1-mini has very likely read the financial commentary describing that outcome. Assigning a high-conviction bullish stance to such a post is then partly hindsight, not a pure text read — the label is contaminated even though no numeric return column was joined. This is a circularity the 'no forward returns in labels' framing does not catch, because the leak is in the labeler's weights, not in the data join.
  - *Fix:* Report a FinBERT-only directional signal as a robustness check for RQ2/RQ3. FinBERT's training predates these events, so any result that survives a FinBERT-only signal is immune to the LLM-cutoff leak. Also disclose gpt-4.1-mini's training cutoff relative to the post dates: if the cutoff predates the 2025-26 confirmatory regime, the concern is confined to lower-stakes pre-inauguration posts (demote to medium); if it postdates the famous episodes, the LLM-labelled results cannot be called text-only. The fix (FinBERT-only robustness column) is the same and cheap regardless.
- **[MEDIUM]** Reporting topic_china<0 as a 'finding' is circular on two levels. (a) Substantively it restates a known prior — China/tariff mentions are risk-off — so a negative coefficient is not a discovery, it's a sanity check that the keyword fires. (b) Technically, in a single tariff-war regime topic_china and topic_tariffs_trade are near-collinear (China is the principal tariff counterparty), so the individual topic_china coefficient is not separately identified; the OLS grid in analyse_asset() reports topic_china and topic_tariffs_trade as distinct terms, which reads structure into a collinear partition. The 'finding' is then an artifact of which collinear regressor absorbs the shared variance.
  - *Fix:* Reframe the contribution as INCREMENTAL: does the signal add timing or magnitude information beyond a trivial 'China/tariff mention = risk-off' keyword baseline, and count it only if it beats that baseline out-of-sample. For the collinearity, report the JOINT effect of the trade/China topic block (e.g. an F-test or a combined risk-off dummy) rather than separate coefficients, and disclose the topic-dummy correlation matrix so readers can see the near-collinearity.
- **[MEDIUM]** The advertised look-ahead safeguard — 'LightGBM with purged + embargoed (24h) walk-forward CV' and Romano-Wolf — is described in the docstring and PREREGISTRATION.md but is NOT implemented in the confirmatory script. _phase2_reaction.py runs an OLS grid with day-clustered SEs + BH-FDR, plus a single final-20%-by-time bootstrap edge test; there is no LightGBM, no PurgedKFold, and no Romano-Wolf anywhere in the executable code (grep across the repo finds these terms only in prose). Consequently (i) the embargo's sufficiency cannot be assessed as written because the CV it embargoes does not run, and (ii) the non-linear robustness check the paper claims is absent. Separately, even if the CV existed, a 24h embargo equal to the max horizon is marginal for the VOLATILITY targets specifically: realized vol is long-memory, so fwd_vol_24 in train stays correlated with fwd_vol_24 in test across a 24h buffer, and both are mechanically linked to the trail_vol_24 control across the train/test boundary. For return targets a horizon-length embargo is defensible (returns are ~serially uncorrelated, so purging the mechanical overlap suffices).
  - *Fix:* Either implement the pre-registered purged+embargoed LightGBM CV and Romano-Wolf, or remove those claims from the manuscript and report only what was run. For the vol regressions specifically, set the embargo to horizon PLUS a fraction (e.g. 1-2% of sample, López de Prado style) to break the long-memory and trail_vol coupling across the boundary; a horizon-only embargo is acceptable for the return-sign targets.

## Multiple testing / researcher degrees of freedom / p-hacking: choice of BH-FDR q=0.10 vs FWER given the asset×horizon×target×term grid; credibility of the Phase-0 (exploratory) / Phase-2 (confirmatory) split given the taxonomy was tuned on observed 2025 episodes; whether the pre-registration is specific enough to bind the analysis.

**What holds up:** Three things are genuinely sound on this axis and should be kept. (1) The RQ3 tradeable-edge test is the one piece of the design that actually binds: a true final-20%-by-calendar-time OOS holdout, a decision rule fixed in advance (block-bootstrap 95% Sharpe-CI lower bound > 0, costs fixed at 0.24% RT), and a pre-stated null-result option ('no tradeable edge after costs' is a valid outcome). Confirmatory weight CAN legitimately rest here even when it cannot rest on RQ1/RQ2. The implementation in _phase2_reaction.py (_edge_stats / oos_trading_edge) matches the pre-reg. (2) The pre-registration is git-timestamp-frozen before Phase-2 results were computed, and the --confirm path refuses to write the paper results file until LLM coverage >= 95% — this is more pre-commitment discipline than most finance event studies show. (3) Phase 0 is honestly labelled exploratory/hypothesis-generating rather than buried, and the 0.92 crypto co-movement is explicitly disclosed (the paper itself says 'one effect measured thrice, not three confirmations'). The intent on the multiplicity problem is correct even where the execution is not. The text-only signal construction (no forward returns in any label) is real but is a look-ahead/circularity matter for a different referee, not this axis.

**Verdict:** Not publication-credible on this axis as written. The headline RQ1/RQ2 'confirmations' are not credibly confirmatory: (a) they re-test on the SAME sample the exact hypotheses Phase 0 generated, including a topic class (market_directive) invented from an episode that sits inside the test window, so the git freeze dates the second look, not the hypothesis; (b) the primary error control is FDR q=0.10 — a lenient discovery rate, wrong for a single confirmatory headline that demands FWER — and the pre-registered fix (Romano-Wolf) is not implemented in code; (c) the FDR family that is actually corrected is a gerrymandered 4-crypto-term subset, while the pre-registered macro panel and the non-linear LightGBM check are silently absent from the confirmatory script with no logged deviation, which directly violates the pre-reg's own binding rule. The design is salvageable: rest confirmatory weight on the genuinely-held-out RQ3 OOS test, demote RQ1/RQ2 to exploratory OR re-confirm them on a true time-split, switch the primary control to FWER, and implement the pre-registered Romano-Wolf so dependence is handled by joint resampling.

- **[HIGH]** The exploratory/confirmatory firewall is not real for RQ1/RQ2: Phase 2 re-tests Phase-0 hypotheses on the SAME ~33k-post sample. Worse, the taxonomy was tuned on observed in-sample episodes — WORKLOG lines 48-53 and the project memory confirm 'market_directive' and 'reassurance' were ADDED after observing the 2025-04-09 'GREAT TIME TO BUY' episode because the keyword taxonomy missed it. That category is then carried into the Phase-2 confirmatory FDR grid (it is one of the four extracted terms in _phase2_reaction.py line 160: 'topic_market_directive'). The pre-registration's git timestamp (2026-06-02) freezes the SECOND look, not the moment of hypothesis generation; the categories, the china-vol and 1-4h-vol effects, and the horizon focus were all chosen after seeing the data. A pre-registration that codifies an already-observed result is a confirmatory veneer over exploratory analysis. This is the single largest threat to the paper's confirmatory claims and it makes every downstream multiple-testing correction cosmetic, because the family was pruned to the survivors before correction was applied (selection-then-test bias / 'freedman's paradox').
  - *Fix:* Make the firewall temporal, not just procedural. Split by calendar time: generate/refine the taxonomy and hypotheses on a pre-2025 window (Phase 0), then confirm RQ1/RQ2 ONLY on a genuinely held-out 2025-26 window that played no role in taxonomy design — and report that the held-out window is dominated by one tariff regime as a power caveat. If a clean time-split is infeasible because the interesting episodes are all in 2025, then explicitly DOWNGRADE RQ1/RQ2 to exploratory and rest all confirmatory weight on RQ3's existing OOS holdout, which is the only test whose sample was truly walled off. For any data-derived category (market_directive, reassurance), either pre-specify it from an external/prior corpus or report it as exploratory with an n-disclosure (market_directive is n=2 — it is a case study and must never sit in a stats grid).
- **[HIGH]** Wrong error rate for a confirmatory headline, at a lenient threshold. RQ1 is asserted as a confirmed real effect ('survives the null'), which is a single confirmatory claim — the field expects family-wise error (FWER) control here, not a false-discovery RATE. FDR is a screening/discovery criterion appropriate to Phase 0, not to a Phase-2 'confirmed' verdict. Compounding this, q is set to 0.10, not 0.05 — a doubled, unjustified tolerance that is itself a researcher degree of freedom. Note: do NOT argue BH is 'invalid' under the 0.92 co-movement — under positive dependence (PRDS) Benjamini-Yekutieli shows BH still controls FDR. The correct objection is the error-rate MISMATCH (FDR where FWER is required) plus the leniency of q=0.10, not BH's validity per se.
  - *Fix:* For the confirmatory RQ1/RQ2 verdicts, control FWER, not FDR — report Holm or, better, the pre-registered Romano-Wolf (see next item) as PRIMARY, with BH-FDR demoted to an exploratory screen. If FDR is retained anywhere, justify q in advance against the cost of a false confirmatory claim and default to q=0.05. State explicitly which single coefficient is the confirmatory test of RQ1 (e.g., the 'signal' or event-dummy beta on fwd_vol_4 for BTC) so the headline rides on one pre-specified test, not on whichever of dozens of cells survives.
- **[HIGH]** The pre-registered dependence remedy is disclosed but NOT implemented, and the implementation contradicts the disclosure in the lenient direction. The pre-reg (section 5) says the three crypto assets are 'treated as ~one effect, not three independent confirmations.' But the code does the opposite: analyse_asset is called in a loop 'for sym in CRYPTO' and every asset's p-value for each term is appended as an independent entry into the single BH pool (_phase2_reaction.py lines 160-166, 236-243). With assets at 0.92 correlation plus overlapping 1/4/24h windows on shared return/vol series, these near-duplicate small p-values cluster at the lowest ranks and mutually EASE each other's BH rejection (the step-up threshold p(i) <= i*q/m is easier to clear when several near-identical small p's co-occur). So the correction is anti-conservative exactly where the paper claims it is conservative, and the disclosed 'one-effect' collapse is never actually performed.
  - *Fix:* Implement the collapse you disclosed. Either (a) average the three crypto assets into a single panel/SUR target and run ONE test per term×horizon, or (b) run a multivariate test that accounts for the cross-asset covariance (stacked GMM / SUR with a joint Wald, or a single panel regression with asset fixed effects and two-way clustered SEs), so each economic hypothesis contributes one p-value to the family. Then apply the correction to the de-duplicated family.
- **[MEDIUM]** The corrected family is gerrymandered, and two pre-registered families are silently missing — which both shrinks m (raising the BH threshold and easing rejection) and means whole blocks of looks dodge correction entirely. Verified in code: (i) the macro panel (DXY, GOLD, SPX, NDX, US10Y, VIX) named in the abstract and pre-reg section 3 appears ONLY in the docstring narration (line 4); main() loops 'for sym in CRYPTO' and never touches macro — so the macro family is unimplemented, not merely uncorrected. (iii) Within the crypto grid, only 4 of the ~14 OLS coefficients per regression are extracted into the pool (line 160: signal, topic_china, topic_tariffs_trade, topic_market_directive); the other 7 topic dummies + novelty + trailing_vol are dropped, and the choice of WHICH 4 is itself a post-hoc degree of freedom that happens to coincide with the Phase-0 winners. Neither the macro drop nor the LightGBM drop is recorded as a deviation in WORKLOG, violating the pre-reg's own rule (lines 11-13) that 'any deviation... must be logged.'
  - *Fix:* Fix the family to its full pre-registered extent and correct over ALL of it: every asset (crypto + macro), every horizon, every target, and every coefficient that is interpreted in the paper — or formally narrow the pre-registration and log the narrowing as a dated deviation BEFORE seeing results. Implement the macro panel and LightGBM as pre-registered, or strike them from the abstract/pre-reg. Report m (the family size) explicitly next to the BH threshold so readers can see the correction was not weakened by under-counting. If only 4 terms are of interest, pre-commit to those 4 with an a-priori economic rationale, not by their Phase-0 performance.
- **[MEDIUM]** The pre-registration is under-specified at exactly the points where degrees of freedom bite, so it does not bind the analysis. It does not fix: which single coefficient is THE confirmatory test of each RQ; the burst aggregation rule (the code uses signal=mean, novelty=max, topic=max within a burst — line 70-81 — an unregistered choice); the 30-min burst window and the hour-anchoring (vs first-minute reaction, acknowledged as 'conservative' but never registered as a sensitivity); the exact null-matching cells (era × trailing-vol QUINTILE — quintile count is a free parameter); and the seed-dependence of the 2000-draw bootstrap. Then re-run and report any change from the loose-spec results as a deviation. Add a sensitivity table over the aggregation/window choices so reviewers can see the headline is not knife-edge on an unregistered knob (the repo memory shows prior strategies that were exactly such knife-edge artifacts).
- **[LOW]** Within-regression multiplicity is unaddressed: each day-clustered OLS estimates ~14 coefficients (signal + novelty + 11 topic dummies + trailing_vol), but inference is presented per-coefficient with no acknowledgement that reading 14 t-stats per regression, across 6 targets and 3 assets, is itself a large implicit search even before the cross-regression FDR. ## NULL DESIGN: the era + trailing-24h-vol-quintile-matched bootstrap, the burst independence unit, and the construction of a defensible placebo for the RQ1 forward-volatility finding.

**What holds up:** Several design choices on this axis are genuinely above the bar for this literature and should be kept. (1) Including a trailing-vol match AT ALL directly targets the correct confound — endogenous posting timing ("he posts because markets are already moving") — which most event studies in this space ignore; the authors clearly understood the threat. (2) trail_vol_24 is computed with .shift(1) (line 119), so it is strictly known at event time — genuinely look-ahead-free, and the quintile bins are cut from the full-era pool so event hours land in well-defined buckets. (3) Excluding the 24h of post-event hours from the null pool (line 136) is the right instinct: it prevents the event's own aftermath from polluting the comparison set. (4) Restricting the null pool to the events' own era (line 141) correctly guards against regime drift (2023-24 bull vs 2025-26 chop) masquerading as an effect. (5) Hour-of-day bucketing is appropriate for crypto, which has real intraday seasonality. (6) The two-sided percentile p-value (lines 171-175) is correctly and conservatively computed (2*min of the two tails, capped at 1). (7) The 0.92 cross-asset co-movement is disclosed and the three assets are explicitly treated as ~one effect rather than three confirmations — honest, even though the null cannot fix that non-independence. The vol effect also corroborates across an independent specification (the all-bars day-clustered OLS event dummy), not the matched-null alone.

**Verdict:** Not yet publication-credible on the null axis as it stands, though the gap is closable. The matched bootstrap correctly controls the FIRST moment of trailing vol across strata, but the headline RQ1 result (forward vol +3-5bp/1-4h, p in 0.007-0.05) hangs on borderline p-values produced by a null whose sampling variance is understated, against an event statistic computed over anchors that are neither serially independent nor evenly spread in time. The most likely failure mode is anti-conservative inference: the true standard error of the event mean is larger than the i.i.d. bootstrap null implies, so p-values near 0.01-0.05 are not trustworthy at face value. With the three concrete fixes below — a stratum balance table, a clustered/block-aware null or time-shift placebo, and a within-regime robustness check — the finding could be made defensible OR shown to be residual vol persistence. As currently constructed the referee cannot tell which.

- **[HIGH]** The bootstrap null understates sampling variance, biasing inference anti-conservatively, precisely where the headline result lives. In null_distribution (lines 161-167) each of the 2000 draws samples n_event individual bar-hours INDEPENDENTLY with replacement within strata and takes the simple mean. This treats the null statistic as a mean of i.i.d. observations. But (a) forward realized vol over 1-4h is strongly autocorrelated bar-to-bar, and (b) the EVENT sample of 1410/658 anchors is not evenly spaced — bursts cluster (multiple per day across the in-office tariff-war regime; the busiest months carry 45-63 bursts each) and their 1-4h forward-vol windows overlap and co-move. The effective number of independent event observations is far below 1410. The i.i.d. resampling produces a null sampling distribution that is too tight (null SD < the true SD of the event mean under serial+cross-event dependence), so the percentile p-value is too small. This directly threatens the marginal headline p-values (BTC 4h p=0.008, 1h p=0.021; ETH p=0.007-0.047; SOL in-office p=0.023-0.033): a correctly-dependence-aware null could push several of these above 0.05, and they are already the survivors after the authors' own controls.
  - *Fix:* Make both sides of the comparison dependence-aware. Replace the i.i.d. within-stratum draw with a block/clustered resample: draw whole calendar-DAY blocks (matching the day-clustered SE choice used elsewhere in the same script) or circular blocks of consecutive hours so the null preserves the autocorrelation of forward vol. Equivalently/additionally, attach a block-bootstrap standard error to the EVENT mean itself (resample event days, not individual bursts) and compare the observed effect to that. Report the ratio of the i.i.d. null SD to the block null SD as a diagnostic — if it is materially >1, the original p-values were optimistic. State the effective sample size (e.g. number of distinct event days) alongside n=1410.
- **[HIGH]** Quintile matching controls only a coarse, level-based summary of trailing vol and likely leaves residual confounding in the tail — and the paper provides no evidence the match actually balanced. Five buckets (lines 146-152) are coarse for a right-skewed variable like realized vol: the top quintile spans a very wide, heavy-tailed range. If posting timing is endogenous, event hours plausibly sit systematically HIGH within their quintile (he posts as vol is climbing), so within-stratum the event hours have higher mean trail_vol_24 than the null draws, and forward vol persistence — not any post effect — produces the +3-5bp gap. The companion OLS 'control' uses trail_vol_24 as a LINEAR level term (line 229), which under-adjusts exactly in the heavy tail where this matters. Nothing in the output demonstrates that event vs matched-null trailing vol is actually balanced.
  - *Fix:* (1) Add a balance table: report mean and a high quantile (e.g. p90) of trail_vol_24 for event hours vs matched-null draws, WITHIN each stratum (or overall after matching). If they are not balanced the match failed and the effect is residual vol persistence. (2) Tighten the match: use continuous caliper / nearest-neighbour matching on trailing vol (or 10 deciles) instead of 5 buckets. (3) In the OLS, enter log(trail_vol_24) or a spline/cubic, not the level, so the control adjusts in the tail. Show the event coefficient is stable across these specifications.
- **[MEDIUM]** The 'burst' (posts <30min apart) is a reasonable de-duplication unit but does NOT establish independence across observations, and the analysis implicitly relies on it for the count of evidence. The independence-relevant horizon is the forward window (up to 24h, with 1-4h being the headline), not 30 minutes. Two bursts 90 minutes or 3 hours apart are separate burst_ids yet share almost entirely overlapping 4h/24h forward windows and the same local vol state — they are not independent draws of the forward-vol process. The de-dup to one obs per anchor HOUR (line 91) only removes within-hour duplicates; it does nothing for the dense multi-burst-per-day clustering that drives the in-office sample. So '1410 bursts' overstates the independent information content, compounding the variance issue above.
  - *Fix:* Decouple the clustering unit for SEs from the 30-min burst-formation rule. For inference, cluster/aggregate at the level of the forward horizon — e.g. collapse to one observation per non-overlapping window (or per event-day) for the headline 4h test, or explicitly model the overlap. At minimum report results both at burst level and aggregated to event-day level; if the effect weakens materially when overlapping windows are collapsed, that is the honest measure of evidence. Also report a leave-one-regime-out / within-regime robustness check: re-estimate excluding the densest tariff-war stretch to show the effect is not carried by one autocorrelated episode.
- **[MEDIUM]** The corpus offers a stronger, dependence-preserving placebo that the design does not use. The current null breaks the link between posts and markets by randomizing WHICH hours are sampled, which is exactly what destroys the event sample's autocorrelation/clustering structure (issue 1). A placebo that instead breaks the link while PRESERVING the events' temporal structure would be far more convincing and is the natural construction the angle calls for.
  - *Fix:* Add a time-shift / circular-block placebo: keep the exact set of event timestamps with their real inter-event spacing and within-burst overlap, then shift ALL anchors by a single random global offset Delta (modulo the era length), rejecting shifts that land on or near true events; recompute the event-mean statistic; repeat to build the null. This preserves the clustering and serial correlation of the event sample (so the null variance is correct) while severing the post->market link. Optionally accept only offsets Delta that preserve vol-quintile balance, or reweight draws to balance trailing vol, to keep the endogenous-timing control. This single addition answers both the variance critique and the 'stronger placebo' question, and is more informative than further stratification tweaks.
- **[LOW]** Pre-event contamination of the null pool is a correctness nit, but note its direction is CONSERVATIVE, not inflationary. The tainted set (line 136, range(25)) excludes only the 24h AFTER each event, not the hours BEFORE. Null-pool hours sitting just before an event have forward windows that overlap the event's own aftermath, dragging the null mean UP toward the event mean. This shrinks the observed gap, so it cannot manufacture the positive result — if anything it makes the test slightly harder to pass.
  - *Fix:* For cleanliness and to remove any ambiguity, embargo the window on BOTH sides: exclude [event - max_horizon, event + 24h] from the null pool (i.e. also blank the forward-horizon hours immediately preceding each event). State explicitly in the paper that the prior behaviour was conservative so reviewers do not misread it as a source of false positives.

## RQ3 trading-rule realism: execution model, every-burst design, holdout power, and the Sharpe-CI decision rule.

**What holds up:** Several things are genuinely sound on this axis. (1) Execution timing is conservative and look-ahead-free, contrary to a naive read of "anchored to the event hour's close." I verified the bar convention: binance_klines.py line 75 indexes bars by open_time (Binance standard), and a post is anchored via ts.floor("h"). So a 10:05 post joins the bar opened at 10:00 whose close prints at 11:00 — strictly AFTER the post. fwd_ret_4 (logc.shift(-4)-logc) is therefore the 11:00->15:00 return, entered at a price that exists after the signal. The rule forgoes the first-minutes reaction (the prereg admits this); that is a cost, not a leak. (2) The decision rule is one-sided and pre-committed: a bootstrap 95% CI lower bound > 0, with "no tradeable edge after costs" accepted in advance as a valid result. This forecloses rule-mining / threshold-hacking and is exactly the discipline most event-trading papers lack. (3) The test is on a true time-ordered final-20% OOS holdout, net of an explicit 0.24% round trip, and the policy/commentary reporting is informative. (4) The harness was written before results were seen and the prereg is git-timestamped. These are the right instincts.

**Verdict:** Not yet publication-credible on the trading-realism axis, though the bones are good and the headline 'no tradeable edge' conclusion is almost certainly robust (so the defects do not threaten a false-positive claim — they threaten the paper's ability to make the test informative). The execution anchoring is genuinely conservative and leak-free (verified open-time bars), the pre-committed one-sided CI bar is disciplined, and the OOS split is honest. But the RQ3 test as designed cannot test the paper's own finding: it trades sign(signal) — the term its OLS shows is directionally insignificant — on every burst, while the only robust directional predictor (topic_market_directive) is rare and largely confined to the training period; gross edge is ~zero, so the loss is not 'mainly costs' so much as 'no signal plus costs.' Compounding this, the 9-cell EDGE-if-any disjunction gets none of the multiplicity control the OLS grid gets (and the subset split is post-prereg), the Sharpe annualization is uninterpretable, the block bootstrap is mis-blocked for an event-spaced overlapping series, slippage is held flat through exactly the high-vol moves the signal predicts, and n~210 in a single regime with the canonical directive event outside the holdout is underpowered. Fixes are concrete and mostly cheap: gate on the directive/conviction term, pre-specify one primary cell with family-wise control, report gross-vs-net and per-trade stats with honest annualization, make slippage vol-dependent, and run a minimum-detectable-effect power calc. With those, RQ3 becomes a credible test of a real hypothesis rather than a near-tautological null on a noise-trading rule.

- **[HIGH]** The traded signal is NOT the signal the paper's own evidence supports, so RQ3 cannot test RQ2/the OLS finding. The rule is sign(signal) on EVERY burst. But the OLS grid shows the continuous `signal` term is directionally insignificant everywhere (every fwd_ret signal coefficient p>0.3; e.g. BTC fwd_ret_4 signal p=0.779, ETH p=0.927). The ONLY robust directional predictor is topic_market_directive (p<0.004 across all assets/horizons, +77bp to +712bp). So the trading rule acts on the term with no demonstrated directional content, diluted across ~210 mostly-noise bursts, while the term that carries the signal (rare directives) is never the trade trigger. The arithmetic confirms there is essentially no gross edge: mean net is within ~1bp of -(24bp cost) in every cell (BTC-all gross +1.0bp, ETH-all -1.8bp, SOL-all +9.8bp). The rule fails for two compounding reasons it conflates: (a) no gross directional edge, and (b) it pays the round-trip toll on all 210 bursts. 'The smoke result lost money mainly to costs' is therefore literally true but misleading: gross is ~zero, so even free execution would not clear the bar.
  - *Fix:* Gate the rule on the carrier of the demonstrated signal: trade only is_market_directive / high-conviction (LLM conviction above a pre-set threshold) bursts, sized by conviction, rather than sign(signal) on every burst. Report gross AND net separately so the reader sees the edge is absent before costs, not merely eaten by them. Crucially, also report the directive-event count in the holdout (see next issue) — the honest finding may be that the only predictive class is too rare OOS to power a tradeable test, which is itself a publishable conclusion and far more defensible than 'no edge from a rule that trades noise.'
- **[HIGH]** The RQ3 verdict is an uncorrected disjunction over 9 correlated cells, with rigor asymmetric to the OLS grid. oos_trading_edge produces {all, policy, commentary} x {BTC,ETH,SOL} and main() declares success via `any_edge |= verdict=='EDGE'` — i.e. EDGE if ANY single cell's ci_low>0. The OLS grid receives BH-FDR (q=0.10) and Romano-Wolf; the trading grid receives no multiplicity control at all. Worse, the policy/commentary split was added AFTER the prereg (logged as advisor-suggested), converting one pre-registered test into a family of nine — post-hoc multiplicity on the confirmatory leg. The crypto assets co-move at r~0.92 (the prereg explicitly refuses to count them as three independent confirmations for OLS, but imposes no such discipline here), so the three 'policy/escalation' cells are nearly the same draw — and they are exactly the borderline ones whose CIs straddle zero ([-46.33,+1.86] BTC, [-38.92,+1.32] ETH, [-27.85,+11.11] SOL). On a re-run these are precisely where a spurious 'EDGE on >=1 asset' could appear.

If a family is reported, apply family-wise control (Romano-Wolf or a Bonferroni-Holm over the effective number of independent cells given the 0.92 correlation, ~3-4 not 9) and report the subset split as exploratory, matching the OLS leg's standard. Replace the `any_edge |= ...` disjunction with the corrected primary-cell decision.
- **[MEDIUM]** The block bootstrap and its annualization are mis-specified for an event-spaced return series, making the reported Sharpes uninterpretable. `net` is one value per OOS burst (sign(signal)*fwd_ret_4 - cost), ordered by event time but NOT equally spaced — bursts cluster on news days and can overlap (two bursts <4h apart share part of the 4h hold). block_bootstrap_sharpe_ci uses block=24 CONTIGUOUS elements: that resamples 24 consecutive EVENTS, not 24 hours, so the block length has no relation to the 4h autocorrelation/overlap structure it is meant to preserve. The annualization sqrt(365*24/4)~93.8 assumes continuous 4h redeployment (2190 trades/yr) when the rule actually fires ~210x/yr with capital idle between — inflating per-trade Sharpe ~-0.09 into the absurd -11.25. Note this scale factor is positive and common to both CI bounds, so it does NOT change the sign of ci_low or the verdict; it is a reporting/interpretability defect, not a decision-validity one. But the block-vs-overlap mismatch IS a validity concern for the CI width.
  - *Fix:* Two fixes. (a) Build a proper equity curve that includes flat/cash periods between events and annualize by sqrt(actual trades per year), or simply report per-trade mean and IQR and a t-stat on the per-trade net return — do not multiply a sparse per-event mean by 93.8. (b) Choose the bootstrap block to span the actual overlap horizon: since holds are 4h and the series is event-indexed, either de-overlap (drop bursts whose 4h windows intersect, or use non-overlapping forward windows) or use a stationary/circular bootstrap with block length tuned to the empirical autocorrelation of the event-spaced net series, and justify it. Report the resulting Sharpe in interpretable units.
- **[MEDIUM]** Flat 2bp/side slippage is implausibly optimistic for exactly the events this study is about. backtester.py hard-codes SLIPPAGE_BPS=0.0002 (2bp) constant regardless of regime. The entire thesis is that Trump bursts (tariff announcements, the April-9 directive) drive 1-4h vol ~+5bp and large directional moves (+166 to +712bp). Trading INTO those moves at the open of the next hour, in the highest-vol minutes for crypto, will incur spread widening and impact far above 2bp — yet cost is modeled as a fixed 24bp that ignores the vol state the signal itself predicts. This biases the net result UP (too favorable), so it does not rescue the null verdict, but it means a future 'edge' claim would be built on an optimistic cost floor, and it understates how hostile execution is in the regime that matters.
  - *Fix:* Make slippage state-dependent: scale per-side slippage with contemporaneous spread or with trailing/realized vol (e.g. slippage_bps = max(2bp, c * fwd_vol_1 or current bid-ask), calibrated to historical crypto book depth around news). Report sensitivity of the RQ3 verdict to 2/5/10bp per side. Since the signal predicts vol, the cost model must be allowed to respond to the same vol.
- **[MEDIUM]** Holdout power and single-regime confound. n=210 per asset (130 policy / 80 commentary) over the final 20% by time — but bursts are ~0.92-correlated across the three assets, so the effective independent sample is closer to ~210 events, not 630, and the policy subset is ~130 highly-overlapping draws. The holdout window is dominated by the 2025-26 tariff war (single regime), and the one episode that exemplifies a tradeable directive — the 2025-04-09 'GREAT TIME TO BUY' post — sits in the training 80%, not the OOS 20%. So the rule is most likely evaluated on a holdout nearly devoid of the only event class with demonstrated edge, while being asked to detect a per-trade Sharpe ~0.09 effect. The CI half-widths (e.g. ~+/-15-20 annualized) confirm the test is underpowered to resolve a small positive edge.
  - *Fix:* Run an ex-ante power calculation: given per-trade net std ~2-3% and ~210 events, what true per-trade edge is detectable at the ci_low>0 bar? State the minimum detectable effect and concede the test cannot resolve plausible edges below it. Strengthen with a purged time-series cross-validated OOS (multiple rolling holdouts) rather than one terminal 20% block, and stratify so the directive class has non-trivial representation in every test fold. Report the count of is_market_directive bursts in the holdout explicitly.
- **[LOW]** The 4h hold and exit are unmodeled relative to the rule's own structure. The trade is sign(signal)-held-exactly-4h with no stop, no path-dependence, and no handling of overlapping positions (a new same-direction burst 1h later is treated as an independent trade rather than a position extension), so the realized cost count (one round trip per burst) can over- or under-state real turnover. Minor relative to the above, but it muddies the 'lost to costs' attribution.
  - *Fix:* Define and pre-register the position-management rule for overlapping bursts (net the exposure, or extend the hold and charge one round trip), and report turnover/number of distinct round trips actually incurred so the cost attribution is exact.

## Cross-asset independence and external validity: whether "3 assets" constitutes genuine replication given BTC/ETH/SOL co-movement (~0.92), whether the single 2025-26 tariff-war regime undermines generalizability, the absence of placebo actors, and regime-dependence of the findings.

**What holds up:** Several things are genuinely sound on this axis and should be preserved. (1) The co-movement is not hidden: the preregistration §5 explicitly discloses BTC/ETH/SOL ~0.92 and states the three are "treated as ~one effect, not three independent confirmations," and the FDR grid is framed accordingly. This is more honest than most cross-asset event studies. (2) The era split (pre-2025 out-of-office vs 2025+ in-office) is a genuine, if confounded, attempt at a generalization/power test rather than a single pooled fit. (3) RQ1's volatility result is NOT purely regime-bound: the repo's own results (docs/TRUMP_EVENT_STUDY.md) show the 4h vol effect surviving the vol-matched null pre-2025 for BTC (p=0.011), so there is real cross-regime evidence for the volatility channel specifically. (4) The RQ3 "no tradeable edge after costs" framing, with a pre-committed Sharpe-CI lower-bound > 0 decision rule, structurally caps over-claiming on the most fragile (directional/regime-bound) part of the study. (5) The 24h vol effect was correctly diagnosed as vol-clustering and discarded, showing the controls actually bite. The instinct to keep cross-asset agreement as a sanity check is reasonable — the problem is using it as a formal robustness gate, not the check itself.

**Verdict:** Not yet publication-credible on the cross-asset / external-validity axis as currently framed, but fixable without new data on most points. The paper's honesty is above average — co-movement and single-regime dominance are disclosed, and the volatility (RQ1) channel has genuine cross-regime support (pre-2025 BTC). The two reject-resubmit-grade gaps are (1) the complete absence of a placebo/negative-control actor, which leaves the headline 'Trump posts predict volatility' unidentified against 'salient-actor posting correlates with vol in a news-heavy regime,' and (2) over-claimed external validity for the directional/topic results, which are inseparable from the tariff-war regime in-sample. Two further medium issues — pseudo-replication with a self-contradicting 2-of-3 replication rule, and an inauguration split that doubles as an unacknowledged regime split — are straightforward to fix by pooling into a single dependence-robust panel and adding an orthogonal regime cut. Recommendation: major revision. Confine the directional claims to the tariff regime, add a placebo actor (or contemporaneous-news control), and replace the three per-asset regressions with one panel estimate; the vol finding can survive, the directional findings should be reframed as regime-conditional.

- **[HIGH]** No placebo / negative-control actor anywhere in the pipeline. The design cannot separate 'Trump's text carries volatility-relevant information' from the weaker, far more mundane claim 'any salient actor's posting activity correlates with crypto volatility during a volatile macro regime.' Posting bursts are endogenous to news flow; on tariff-war days both Trump's posting rate AND realized vol rise because the same exogenous policy events drive both. The trailing-24h-vol-quintile match controls for the level of vol clustering already underway, but it does NOT control for the arrival of contemporaneous news that elevates BOTH posting and forward vol. A grep of the entire repo confirms 'placebo actors' appears only as future work in docs/PAPER/OUTLINE.md and THESIS_STRUCTURE.md; it is implemented nowhere. The ~+5bp/4h vol bump is therefore not attributable to Trump specifically.
  - *Fix:* Run the identical burst pipeline (same burst-definition, same hour anchoring, same vol-matched null, same era) on one or more negative-control feeds posting over the same hours: a non-Trump high-salience political/financial account (e.g., a Treasury/Fed communications feed, a major newswire headline feed, or a matched high-engagement non-policy account). The Trump vol bump 'counts' only to the extent it exceeds the placebo feed's bump on the same bars. If a clean actor feed is unobtainable, at minimum add a contemporaneous-news control (e.g., a macro-surprise / scheduled-announcement dummy, or count of same-hour Reuters/Bloomberg crypto headlines) to the vol regression and show the event dummy survives it. Report this as a falsification test, not a footnote.
- **[HIGH]** External validity is structurally capped by single-regime dominance, and this bites the DIRECTIONAL claims hardest. The market-relevant bursts cluster heavily in the 2025-26 tariff war (the paper concedes 'single regime dominated by the 2025-26 tariff war' and 'heavy April-2025 clustering'). The topic-direction findings — china/tariffs -> negative next-hour return (china 1h: BTC -8.7bp, ETH -12.1bp, SOL -17.9bp) — are by construction tariff-war content. There is no out-of-regime sample in which 'china/tariff post -> negative return' can be tested, so the claim that the directional signal generalizes is untestable with this data, not merely under-powered. The famous n=2 'GREAT TIME TO BUY' directive is the extreme case: it is one episode, not a population.
  - *Fix:* Sharpen the abstract/conclusion to state the external-validity ceiling explicitly: the directional/topic results are conditional on the tariff-war regime and the design cannot establish generalization because market-moving Trump events do not occur outside it in-sample. Restrict directional claims to 'within the 2025-26 tariff regime' and remove any language implying out-of-sample directional generality. For RQ2/RQ3, report the in-office/tariff subsample as the population of interest and explicitly flag that any deployed directional rule is a bet that the regime persists. Keep RQ1 (vol) separable since it has the pre-2025 BTC support.
- **[MEDIUM]** Pseudo-replication and a self-contradicting replication rule. _event_study_trump.py lines 18-20 set a 'replication bar' that a result counts only if it holds on 'at least 2 of 3 assets,' and the narrative leans on cross-asset agreement ('same sign all assets,' 'most consistent across assets'). But the preregistration §5 simultaneously states the three assets are ~one factor (rho 0.92) and 'not three independent confirmations.' These two rules contradict: you cannot both treat the assets as one effect AND use 2-of-3 cross-asset agreement as a robustness gate. Running three separate per-asset OLS (confirmed in both _event_study_trump.py and _phase2_reaction.py) and counting them as three tests inflates the effective test count feeding the BH-FDR grid and the 'N of M survive' framing. Report one effective test per (horizon x target x term), not three, and feed that reduced grid to BH-FDR. Delete the '2-of-3 assets' replication bar or relabel it as a within-factor consistency diagnostic with no inferential weight. This also de-duplicates the FDR grid honestly.
- **[MEDIUM]** The inauguration split conflates a power/time split with a regime split, so it cannot do the external-validity work it is implicitly asked to do. SPLIT_AT = 2025-01-20 separates 'out of office' from 'in office,' but 'in office' is nearly coextensive with the 2025-26 tariff war. The repo's own findings note this ('in office coincides with the 2025 tariff-war regime, so the split is not a clean power-effect isolator'). Any 'effect is stronger in office' inference is therefore indistinguishable from 'effect is stronger during the tariff regime' or 'effect is stronger when posting volume and policy salience are both high.' Cross-regime stability is asserted via this split but not actually identified by it.
  - *Fix:* Add an orthogonal regime axis independent of the office split: e.g., partition by a macro-vol regime (VIX or BTC-realized-vol tercile), by tariff-headline intensity, or by a non-tariff vs tariff topic cut WITHIN the in-office period, and show whether the vol effect persists across those cuts. Report the office split and the regime split as separate robustness columns rather than letting one proxy for the other. Where they cannot be disentangled in-sample, say so.

## Signal construction and reproducibility: the mixed FinBERT/LLM labelling seam, gpt-4.1-mini label determinism, stance-vs-market-impact-direction ambiguity, and whether the blended `signal` is a coherent regressor for RQ2.

**What holds up:** Several design choices on this axis are genuinely sound and should be kept. (1) The hardest reproducibility trap is correctly avoided: no forward returns enter any label — FinBERT, the keyword taxonomy, the LLM prompt (explicitly instructed to ignore hindsight), and the causal trailing-7d novelty are all text-only, so the labels are not circular with the targets. (2) The LLM call uses temperature=0 and a structured JSON schema, captures token usage, and is checkpointed/resumable per-id — the right engineering instincts even if not sufficient for determinism. (3) The keyword taxonomy and burst definition are frozen, rule-based, and dependency-light, so those components are exactly reproducible. (4) Raw component columns (finbert_score, llm_stance, conviction, topic dummies, novelty) are all persisted alongside the blend, so every fix below can be run on the existing artifact without re-labelling. (5) The authors already self-document the stance-vs-impact tension in trump_signal.py and demote their own validation gate to a 'build-consistency smoke test, NOT evidence of skill' — the honesty is real and the raw materials for proper validation are on disk. (6) Critically, RQ1 does not depend on the blended signal at all, so the headline vol result is not contaminated by this axis.

**Verdict:** Not publication-credible as currently constructed on this axis. RQ1 (the vol finding) is insulated — it runs off trump_events.parquet (topic dummies + FinBERT) and never touches the blended `signal`, so it can stand. But RQ2/RQ3 regress forward returns on a `signal` column that concatenates two labellers measuring different constructs on different scales (verified in the built artifact: LLM branch mean +0.042/std 0.277/39% exactly zero vs FinBERT branch mean -0.187/std 0.513/0% zero — opposite signs), produced by a non-deterministic, version-unpinned LLM call, with a seam whose mixing proportion is build-date-dependent. As written, `signal` is not a fixed, reproducible, or coherent single regressor, so the RQ2 OLS coefficient on `signal` and the RQ3 sign(signal) rule are uninterpretable. This is fixable with the validations below, but the directional claims cannot be accepted until they are run.

- **[HIGH]** LLM labels are non-deterministic and unreproducible by construction. trump_signal_llm.py sets temperature=0 but passes no `seed`, captures no `logprobs`, and pins no model snapshot — MODEL defaults to the moving alias 'gpt-4.1-mini' (line 44). OpenAI chat completions are not deterministic at temperature=0 (MoE routing / batch nondeterminism), and the underlying weights behind a bare alias change silently over time. The ~1,640 LLM stance labels that drive RQ2/RQ3 therefore cannot be regenerated by a referee or by the authors themselves; a re-run could flip stance signs and the is_market_directive/is_policy_signal flags that define the RQ3 subsets. No working paper claim that rests on these labels is reproducible.
  - *Fix:* Re-label against a pinned dated snapshot (e.g. gpt-4.1-mini-2025-XX-XX, recorded in the run manifest), set seed and request logprobs/top_logprobs, and persist the model version, system_fingerprint, prompt hash, and per-label token logprobs into the JSONL. Then quantify label stability: re-run the full market-relevant set 3-5 times and report the per-post stance modal-agreement rate and the fraction of bursts whose sign(signal) flips across runs. Release the exact prompt, the snapshot id, and the resulting label file as a frozen supplement so the labels are auditable even if the API later drifts.
- **[HIGH]** The blended `signal` is not a coherent single construct for RQ2 because its two branches measure different things. The SYSTEM prompt asks the LLM for *market impact direction* ('likely SHORT-TERM effect on risk assets'), whereas the FinBERT fallback scores *rhetorical tone* (P(pos)-P(neg) of the text). These are concatenated into one column via np.where(has_llm, (stance/2)*conviction, finbert_score) (trump_signal.py:67-73). I verified on the built artifact that the two branches have opposite-signed means (LLM +0.042 vs FinBERT -0.187) and that, restricted to posts where BOTH labellers commit a non-neutral call, they agree on direction only 72.7% of the time (n=807) — i.e. ~27% outright sign disagreement on committed labels. A regressor that is sometimes impact-direction and sometimes tone, with branches disagreeing on a quarter of overlapping cases, has no single interpretation; the RQ2 OLS coefficient on `signal` is a weighted average of two incompatible quantities and is not interpretable as 'the effect of the signal'.
  - *Fix:* Do not blend impact and tone into one column. Either (a) define `signal` from a single construct (LLM impact stance only) and treat FinBERT tone as a separate covariate, or (b) keep them separate and run RQ2 with finbert_score and llm_impact as distinct regressors so each coefficient is interpretable. Re-state the RQ2 hypothesis on whichever construct the paper actually claims to test. Report the full FinBERT-vs-LLM concordance table on the overlap (confusion matrix of stance buckets vs finbert sign, with the 72.7% committed-agreement / 27% disagreement figure), so referees can see how often the fallback contradicts the primary labeller.
- **[HIGH]** The seam location — i.e. WHICH posts are LLM-labelled vs FinBERT-labelled — is build-date-dependent and silent, which is a direct reproducibility failure for RQ2/RQ3. The LLM layer is budget-throttled (240k tokens/day), resumable, and appends across days; merge() in trump_signal.py falls back to finbert_score for any post lacking an LLM row, with no flag in `signal` itself marking which branch produced a given value. In the current build, market-relevant coverage is 88.6% — so 11.4% of posts are FinBERT-fallback — but a rebuild on a later day shifts that fraction. Because the fallback subset is systematically different (the FinBERT branch is more bearish, mean -0.187, and ~2x higher signal variance, std 0.513 vs 0.277), the proportion and identity of fallback posts materially changes the regressor. Two honest rebuilds of `signal` on different dates yield different RQ2 inputs. Compounding this, load_bursts() averages `signal` across posts within a burst (signal:'mean'), so a single burst can mix LLM and FinBERT values that live on different scales, and the per-burst regressor depends on labelling order/timing.
  - *Fix:* Freeze the label set: complete LLM coverage to 100% of market-relevant non-noise posts (or to a pre-declared threshold) BEFORE any confirmatory run, snapshot trump_signal.parquet to an immutable, hashed file, and report results only against that frozen artifact. Record per-row provenance (which labeller set each `signal` value) and carry a `signal_source` column into the burst panel. Run the pre-registered RQ2/RQ3 as a sensitivity analysis on (i) LLM-only bursts, (ii) blended bursts, and (iii) bursts excluding any FinBERT-fallback post, to demonstrate the 11% fallback subset is not driving the result. For burst aggregation, aggregate within-source (e.g. conviction-weighted mean of LLM stances) rather than averaging across the seam.
- **[MEDIUM]** Stance-vs-market-impact ambiguity is real and self-acknowledged but not resolved, and it biases the directional sign of the very class the paper highlights. The note at trump_signal.py:88-90 concedes that escalation/'gloating' posts about already-known tariffs receive a positive stance even though the market reaction is negative, and that the bearish reaction 'lives in the topic dummy, not the stance.' If true, then for an important slice of posts sign(signal) is systematically WRONG relative to the realized move, which directly attacks the RQ3 sign(signal) trading rule. The RQ3 escalation/'policy/escalation' subset — flagged as the candidate incremental-information class — is exactly where this mislabelling concentrates.
  - *Fix:* Make the LLM target unambiguous: relabel to predict the directional market REACTION a trader should take (already the stated intent) and explicitly instruct that triumphant framing of bad-for-markets policy (e.g. new tariffs) should read bearish, with few-shot examples of the gloating-but-bearish pattern. Validate by reporting, on a held-out human-audited sample, the rate at which 'triumphant escalation' posts are labelled with the correct reaction sign. Until resolved, report RQ3 with stance-derived sign AND with a topic-dummy-derived sign so the referee can see how sensitive the edge is to the construct choice.
- **[MEDIUM]** The only validation of label quality is the authors' own self-described 'near-tautological build-consistency smoke test.' The _validate gate's single PASS/FAIL checks that the Apr-9 'GREAT TIME TO BUY' post is caught by a class built specifically to catch it; checks [2]/[3] are explicitly descriptive. There is no measurement of whether the LLM stance labels are CORRECT or reproducible across human judges — i.e. no external validity evidence for the construct that RQ2/RQ3 depend on. A referee cannot accept a directional signal whose labels have never been benchmarked against an independent standard.
  - *Fix:* Draw a stratified random sample (e.g. 200-300 market-relevant posts, stratified by topic and by stance bucket), have 2+ independent human annotators label stance/directive/policy under the same instructions, and report inter-annotator agreement (Cohen's/Fleiss' kappa) and LLM-vs-human agreement (kappa + confusion matrix). Oversample the directive/reassurance and escalation classes since those drive RQ3. This is the standard validation a quant-finance referee will demand for any text-derived label, and the raw posts are already on disk to do it.
- **[LOW]** RQ2 and RQ3 are run on different effective populations because of how zeros and the abs()>1e-6 filter interact with the seam. 39.3% of LLM-branch `signal` values are exactly zero (neutral stance OR conviction 0), while 0% of FinBERT-branch values are zero. The RQ2 OLS includes all bursts, but the RQ3 _edge_stats filter (signal.abs()>1e-6) drops the LLM-neutral mass, so the tradeable universe is disproportionately FinBERT-fallback posts plus high-conviction LLM posts. The 'no tradeable edge / edge' RQ3 conclusion is therefore drawn on a different, seam-skewed sample than the RQ2 association test, undermining the narrative that RQ3 operationalizes RQ2.
  - *Fix:* State explicitly the n and the LLM/FinBERT composition of the RQ3 trade set vs the RQ2 regression set, and reconcile them: either include the zero-signal (no-trade) bursts in RQ3 as flat positions so the populations match, or restrict RQ2 to the same non-zero, tradeable bursts. Report the conviction distribution of traded vs non-traded events so the reader can see what the rule actually trades.
